Docker Network type

Bryant Tseng
2021-09-01
docker

background

Docker provides various kinds of network drivers, which can provide different functionality for your docker container.

Here we talk about some of the common network drivers which exist by default.

Network drivers

bridge

  • Default network driver
  • Bridge networks are usually used when your applications run in standalone containers that need to communicate

A default bridge network is created when you start Docker, a newly created container would automatically connect to this network if no other network is specified.

Also a user-defined bridge can be created for additional functionalities.

User-defined bridge network

  • automatic DNS resolution between containers

    Containers within default bridge network can only access each other by using ip-address(unless using --link option).

    In user-defined bridge network, containers can resolve each other by container-name or alias.

  • User-defined bridges provide better isolation.

  • Containers can be attached and detached from user-defined networks on the fly.

  • Each user-defined network creates a configurable bridge.

  • Linked containers on the default bridge network share environment variables.

host

Host mode networking can be useful to optimize performance, and in situations where a container needs to handle a large range of ports, as it does not require network address translation (NAT), and no “userland-proxy” is created for each port.

  • container’s network stack is not isolated from the Docker host (the container shares the host’s networking namespace)
  • container does not get its own IP-address
  • port-mapping does not take effect, and the -p, --publish, -P, and --publish-all option are ignored
